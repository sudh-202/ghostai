## Architecture Decisions

- Kept all theme customization in app-level files and left generated `components/ui/*` files untouched after installation, matching the feature spec.
- Used local/system font stacks instead of remote Google font fetching so the Next.js production build can succeed reliably in this environment.
- Clerk is the active authentication provider, mounted globally in `app/layout.tsx`, themed with the bundled `@clerk/ui` package, and enforced through `proxy.ts`.
- Clerk public route paths are normalized from `NEXT_PUBLIC_CLERK_SIGN_IN_URL` and `NEXT_PUBLIC_CLERK_SIGN_UP_URL` so redirects and route protection use one source of truth.
- Collaborator access remains email-based in Prisma, and Clerk Backend API lookups are used only to enrich collaborator presentation data with display names and avatar URLs at read time.
- Liveblocks server access is created lazily from `LIVEBLOCKS_SECRET_KEY` to keep Next.js builds safe when the secret is missing at import time, and room authorization is always scoped to project IDs.
- The base workspace canvas now uses a typed React Flow + Liveblocks storage model under the `flow` room key, with client-side room bootstrap isolated to `components/editor/liveblocks-canvas.tsx` while the editor route itself stays server-rendered.
- New canvas nodes are inserted through React Flow's change pipeline so drag-to-create shape drops stay synchronized through the same Liveblocks-backed node storage model as future direct node edits.
- Canvas shape rendering now mixes CSS and SVG primitives inside the same `canvasNode` renderer so simple rounded shapes stay lightweight while geometric and cylindrical shapes scale cleanly with arbitrary node sizes.
- Node resize changes continue to flow through React Flow's shared node-change pipeline into Liveblocks storage, while inline label edits write directly to the same typed `flow.nodes[*].data.label` collaborative storage path used by the custom canvas renderer.
- Canvas node theming is now modeled as a predefined collaborative background/text pair in node `data`, keeping swatch selection deterministic and avoiding any freeform color-picker state.
- Canvas edges now use a dedicated custom edge type with collaborative label data in `edge.data.label`, while new connections are still inserted through the shared React Flow/Liveblocks edge-change pipeline.
- Starter-template imports now use predefined typed node/edge graphs and replace the shared `flow` room state in one collaborative mutation before the canvas fits to the imported diagram.
- Canvas participant presence is now scoped to the editor room view: collaborator avatars are rendered from Liveblocks user info inside the canvas overlay, while cursor positions are synchronized through the shared `Presence.cursor` field and the current user remains represented by Clerk's `UserButton`.
- The AI sidebar shell is now isolated in its own workspace component, with open/close state controlled by `WorkspaceShell` and all chat/spec placeholder UI encapsulated locally so future backend AI wiring can attach without reshaping the main editor layout.
- Trigger.dev design-agent runs are now tracked locally in Prisma via `TaskRun`, allowing the API to verify run ownership before minting a public token that is scoped down to a single run.
- The backend design-agent now treats Gemini output as a validated incremental action plan against the existing Liveblocks `flow` graph, keeping collaborative canvas mutations, ephemeral AI presence, and room status messages inside the same realtime system rather than creating parallel task state.
- Liveblocks room providers are now hoisted to workspace level via `LiveblocksWorkspaceProvider` so the AI sidebar and canvas share one room context; `LiveblocksCanvas` now assumes it is inside the provider and only renders the `ClientSideSuspense`+canvas flow tree.
- `ai-chat` feed is kept strictly separate from `ai-status-feed`; `FeedMessageData` is a discriminated union on `kind` so both feeds share the same Liveblocks global type while validators ensure only valid payloads are rendered.
- ProjectSpec metadata and content are now split intentionally: the list endpoint exposes only `id`, `filename`, and `createdAt`, while a separate authenticated content route resolves the private Blob-backed Markdown for preview and the download route remains attachment-only.
